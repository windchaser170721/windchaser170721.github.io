提案一：把流式视频的 KV 压缩技术系统性地引入 VLA ​

一句话总结 ​

机器人长程任务需要"记忆"，而视频大模型那边早已发展出成熟的"长视频记忆压缩"工具，但目前还没人把它们用在机器人上。 我们打算把这些技术直接搬过来，在专门考察记忆能力的机器人 benchmark 上做系统对比，回答"到底哪种机制最有效、掉点曲线长什么样"。

背景：VLA 的"短记忆病" ​

VLA（Vision-Language-Action model） 是当前机器人操作的主流范式：一个大型视觉-语言模型（类似 GPT-4V）外接一个动作输出头，输入图像 + 自然语言指令，直接输出机器人关节命令。代表性工作有 OpenVLA、π0、π0.5（这些都开源）。

但它们绝大多数只看当前一帧做决策。这在两类任务上会失败：

  • 非马尔科夫任务：例如"把所有按钮按一遍"。按下前后画面几乎一样，单帧根本判断不出按了几个。
  • 长程组装/整理任务：需要记住此前做过什么、物体放过哪里。

最朴素的解法（把最近 8 帧拼起来一起喂给 VLA）有两个问题：(1) Transformer 注意力的计算量随帧数二次增长；(2) VLA 预训练数据基本是单帧的，多帧输入反而让模型分布漂移、性能下降。

一个被忽视的现成解决方案 ​

另一个独立发展的研究领域 ——流式视频大模型（处理 1 小时直播流的视觉助手）—— 过去一年发展出了一整套成熟的工具，专门解决"长视频如何高效记忆"。核心机制叫 KV cache 压缩：

简单解释：Transformer 在处理每个 token 时会缓存它的 Key 和 Value 向量供后续步骤复用，这个缓存叫 KV cache。视频特别长时这个缓存会爆炸式增长。KV 压缩就是只保留"重要的" KV、丢掉冗余的，用一个固定大小的"记忆库"来代替无限增长的 cache。

代表性工作（都开源、都是 training-free 或轻量微调）：

  • LiveVLM（2025）：维护一个短期滑窗 + 一个长期记忆库，新问题来了从长期库里检索相关片段。
  • StreamMem（2025）：查询无关的压缩——不知道未来要回答什么问题，就用通用打分把"显著的"视觉 token 留下来。固定内存上限。
  • StreamKV（2025）：基于语义边界的动态分段 + 检索式压缩，效果最好但实现也最复杂。

这些方法在长视频问答上能把 KV 压到原来的 10%–30% 而几乎不掉点。但它们都只在 VideoQA 任务上验证过，从没人在机器人上跑过。

核心想法 ​

把上述三种机制作为"历史压缩模块"直接接到 VLA 主干上，在专门设计来考察记忆能力的机器人 benchmark 上做对照实验。

这是一个低风险、高确定性的研究：所有组件都开源、benchmark 也都开源、不需要造任何新东西。但产出的 insights 会很扎实——目前整个机器人社区缺一份系统的"记忆机制怎么选"的对比研究。

具体方案 ​

Backbone（VLA 主体） ​

选两个有代表性的开源 VLA：

  • OpenVLA-OFT：自回归式，KV cache 行为标准；
  • π0 / π0.5：flow-matching 式（一种类似扩散模型的动作生成方法），结构更复杂，覆盖另一类典型设计。

对比的方法（按抽象程度排序） ​

  1. 单帧 baseline；
  2. 朴素多帧（K=8 帧拼接，二次复杂度 baseline）；
  3. ContextVLA（已有的 amortized 多帧压缩）；
  4. KV-Efficient VLA（机器人侧已有的 RNN-gated 压缩）；
  5. LiveVLM（直接搬）；
  6. StreamMem（直接搬）；
  7. StreamKV（直接搬）；
  8. MemoryVLA 作为 "ceiling" 参考（不计入压缩组，因为它内存不受限）。

Benchmark 矩阵 ​

这是整个工作最关键的部分，必须区分两类任务：

类别 A：长程但仍 Markovian（信息一直可见）

  • LIBERO-Long、SIMPLER-Bridge、RoboCasa
  • 这一类测速度——理论上所有压缩方法都不应该掉点，差距在延迟和内存。

类别 B：真正 Non-Markovian（关键线索已经看不见了，必须靠记忆）

  • MIKASA-Robo（32 个专门的记忆任务）
  • LIBERO-Mem、RMBench、RoboMME（2025–2026 新出的 benchmark）
  • 这一类测记忆质量——压缩比上去后，谁能撑住。

三个核心实验 ​

  1. 压缩比 vs 成功率曲线：保留 100% → 5% 的 KV 时各方法的成功率衰减形状。
  2. 延迟 vs 成功率帕累托前沿：在动态任务（Kinetix）上画。
  3. 按记忆类型分类的 ablation：把任务按"时间记忆 / 空间记忆 / 物体记忆 / 程序记忆"四类拆开，看哪种压缩擅长哪种记忆。

几个工程坑（提前预警） ​

  • 动作 token 不能压：只压视觉 token，本体感觉和动作的 token 数量少、维度低，全部保留。
  • 查询代理的选择：StreamMem 用通用 token 当代理；在 VLA 里最自然的代理是"语言指令 token + 当前观测 token"。
  • Chunk 边界一致性：π0 这类 flow-matching 模型在一个动作 chunk 内部需要稳定的视觉条件。压缩必须对齐到 chunk 边界，不能在 chunk 中间触发。
  • 训练分布漂移：VLA 训练数据多是短轨迹。直接跑长程任务可能因为 context 太长而退化，需要在长程仿真上做轻量 fine-tuning。

预期贡献 ​

  1. 第一份系统对比：流式视频 KV 压缩在机器人记忆任务上的系统评测。
  2. 机制层面的 insights：哪种压缩对哪种记忆类型友好（例如：StreamMem 偏空间记忆、KV-Efficient VLA 偏程序记忆，需要验证）。
  3. 可直接复用的工具：开源所有适配代码，其他 VLA 研究者可以零成本接入这些机制。

资源与时间 ​

  • 代码：所有组件都开源，工程整合约 3–4 周。
  • 算力：仿真 evaluation 单台 A100 即可；fine-tuning 4×A100 一组对照一周内跑完。
  • 总时间：3 个月内出完整 paper。
  • 主要风险：OpenVLA-OFT 单帧设计可能让差距看不出来 → 用 ContextVLA 改造版本作为 fallback。

投稿目标 ​

NeurIPS / CoRL / ICRA。"系统性 evaluation + 跨领域技术迁移"是这几个会比较喜欢的类型。

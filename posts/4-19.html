详细展开昨天说的第一点 ​

3D 几何到底该在策略里从哪里进、以什么频率进、用什么表示进。几何信息从哪里来、注入到 policy 的哪一层、解决的主要瓶颈是什么、代价是什么。VLA 里真正缺的到底是什么，3D 几何应该放在模型的哪个位置，才能既提升操作，又不破坏原有的语言—视觉—动作能力。这几篇论文分别代表了六种不同的答案：把几何放在输入表征里、放在动作离散化里、放在动作专家中间层里、放在视觉骨干里、放在动作头里，或者干脆放在外部慢系统 + 内部快系统的异步接口里。

SpatialVLA ​

VLA 本身缺空间感。 它做了两件基础的事。第一件是 Ego3D Position Encoding，把 3D 信息注入输入观察；第二件是 Adaptive Action Grids，把机器人动作改写成自适应离散的空间动作网格，而不是只把动作看成普通连续控制量。它还在 110 万条真实机器人轨迹上做了预训练，并把 action grids 设计成能在新机器人或新设置上重新离散化，用于跨平台迁移。SpatialVLA 的真正创新点是把空间表征和空间动作表征同时提升到基础建模对象，如果动作发生在 3D 世界里，那么 observation token 和 action token 都应该带有空间结构。

SpatialVLA 的思想：既然操作本质是空间问题，那就从最上游重写输入和动作的表示。这种思路的优点是很干净，因为它不需要依赖一个外部几何 foundation model，也不把几何当成外挂模块；空间性直接刻进了 tokenization。缺点也来自这里：一旦你从输入和动作编码层面重写表示，模型的整个训练和迁移流程都会比较重，而且它更像一个“从头定义 spatial-aware VLA”的方案，不太像给现有 OpenVLA 一类模型直接打补丁。它告诉你空间结构必须进入 VLA，但还没有把“如何最省成本、最少破坏地注入 3D”做到极致。

PointVLA ​

怎么给已经训练好的 2D VLA 无痛补上 3D 世界知识。 出发点：大规模 VLA 已经吃了海量 2D 预训练数据，你不可能每次想加 3D，就把整套 backbone 重训一遍，于是提出一个很工程的方法：冻结原始 action expert，只通过轻量 modular block 把 point cloud 特征注进去；为了不破坏已有表征，它还做了 skip-block analysis，专门找出 action expert 里 相对没那么关键 的 block，把 3D 信息注到这些位置。它报告的几个现象也很说明问题，比如四任务各 20 个 demonstration 的 few-shot multi-tasking、区分真实物体与照片、以及对训练中没见过的桌面高度的适应。

PointVLA 的思想：本质上是一个参数干预理论：不是所有层都值得接 3D，甚至大部分层都不该接。它隐含的观点其实很强——原有 2D VLA 并不是什么都缺，它缺的是在动作生成阶段可用的空间判别力。 所以 PointVLA 选择不碰视觉—语言主干，而是精准地干预 action expert。3D 是一种高价值但高扰动的信号，应该只在最需要的地方进入。

GeoAware-VLA ​

很多 VLA 在新相机视角下失效，不是因为动作学不会，而是因为从 2D 图像里学 3D 一致性太难，视觉编码器本身几何敏感度不够。 它直接换掉或增强视觉骨干的几何能力：用一个冻结的、预训练好的几何视觉模型做 feature extractor，再加一个轻量 projection 层把这些 feature 适配给 policy decoder。

GeoAware-VLA 的思想：它把 3D 问题重新表述成 viewpoint invariance 问题。很多 manipulation 失败表面上看像 grasp 错了、move 错了，实际上根源是视觉表征对视角变化不稳。有时候不需要在 policy 末端拼命补几何，只要前端视觉骨干先把几何一致性学好，后面策略自然轻松很多。它和 PointVLA 形成了很漂亮的对照：PointVLA 认为 3D 应该进 action expert，GeoAware-VLA 认为 3D 应该进 vision backbone。3D 接口 是这个方向的核心变量。

FALCON ​

于是它给出的答案是：把 rich 3D spatial tokens 直接送到 action head，而不是拼到 vision-language backbone。 前面几类方法有两个问题：一种依赖专门传感器，跨模态转移差；另一种把弱几何 cue 粗暴拼进 backbone，结果反而伤到 vision-language alignment。它利用 spatial foundation models 从 RGB 中抽取几何先验，又设计了 Embodied Spatial Model，在有需要时还能融合 depth 或 pose；这些空间 token 最终由一个专门的 Spatial-Enhanced Action Head 消化，而不是与原始语言视觉 token 无差别混合。论文在 3 个仿真 benchmark 和 11 个真实任务上报告了强结果，并强调在 clutter、spatial-prompt conditioning、物体尺度和高度变化下都更稳。

FALCON 的思想：语言推理和几何推理虽然都重要，但它们不一定要在同一个 token 空间里无差别融合。很多早期多模态工作默认 统一 token space = 更强融合 ，但 FALCON 其实反过来说，VLA 里空间信息最该服务的是 action grounding，而不是去扰动原本已经很强的语言—视觉对齐。因此它把几何信息延后到 action head 再发挥作用。这种结构性克制和 backbone 早融合不同，让几何服务动作，而不是让几何接管全部表征。

3D-Mix ​

它做了一个九种 VGGT 融合方案的 pilot study，发现最好的是 semantic-conditioned gated fusion ——也就是根据任务上下文，自适应地平衡 2D 语义特征和 3D 几何特征。它提出 3D-Mix 这个 plug-and-play 模块，能兼容 GR00T-style 和 π-style VLA。

3D-Mix 的思想：除了 gated fusion，它思考3D 语义混合应不应该受任务上下文控制。3D cue 不是每时每刻都同样重要：抓取前接近物体时，几何和尺度很重要；高层语义决策时，语言任务意图更重要。3D-Mix 承认：2D 语义和 3D 几何不是静态加权关系，而是动态竞争关系。谁该主导，应该由当前任务上下文决定。这个思想可以扩展成长期项目，可以继续追问：这个 gating 能否由 contact stage、subgoal phase、uncertainty 或 scene dynamics 来控制，而不只是由静态 token 交互来学。

R3DP ​

可以很聪明地接入 3D foundation model，但闭环控制不能等它慢吞吞推理。 R3DP 的关注点是 几何信息该以什么时间频率进入系统。 它提出一个 asynchronous fast-slow collaboration 结构：重的 3D 模型只在稀疏关键帧上运行，作为 slow system；中间帧则由轻量的 Temporal Feature Prediction Network, TFPNet 预测几何特征，以维持实时控制；同时它还引入 MVFF，在多视角情况下显式使用相机内外参来做 feature 融合。相比单视角和多视角 diffusion policy 基线，平均成功率分别提升 32.9% 和 51.4%；相较 naive 的 DP+VGGT 直接拼接，它的推理时间降低 44.8%。

R3DP 的思想：它把 3D 接口问题 从网络结构继续推进到了系统时序结构。不可能指望一个大几何模型每 30Hz 都完整跑一遍，把大模型变成稀疏校正器，把小模型变成连续补全器。

演化脉络 ​

SpatialVLA 关心的是把空间性刻进输入和动作；PointVLA 关心的是最小代价地把 3D 插入现有 VLA 的 action expert；GeoAware-VLA 关心的是从视觉骨干层面补足几何一致性；FALCON 关心的是让几何在 action head 而不是 backbone 发挥作用；3D-Mix 开始系统比较并提出按上下文动态控制 2D/3D 融合；R3DP 则进一步把问题推进到实时性和异步更新。也就是说，这条线并不是重复劳动，而是在不断逼近一个更完整的问题：几何先验应该以什么表示、在什么层级、以什么频率，进入机器人动作决策。

idea ​

第一，3D 不是普通附加模态，而是结构化约束。这些工作越往后，越少有人把 3D 当“再多一组 feature”。不管是 GeoAware-VLA 的 viewpoint invariance，FALCON 的 spatial-to-action，还是 3D-Mix 的 gated fusion，本质上都在说明：几何的作用是给策略一个关于物体位置、尺度、视角一致性、空间关系的结构约束，而不是多一个传感器而已。

第二，几何最该服务的是动作 grounding，而不是全局表征统一。这点在 PointVLA 和 FALCON 身上体现得最明显。PointVLA 有意识地冻结原 action expert，再把 3D 注入少数 block；FALCON 更明确地把 spatial tokens 交给 Spatial-Enhanced Action Head。这说明当前更有效的范式，往往不是让 3D 从头接管视觉语言主干，而是让它在动作决策的最后几步里提供关键判别力。

第三，3D 接入不是一次性设计，而是一个层级化、时序化接口问题。SpatialVLA 讨论输入和动作空间，GeoAware-VLA 讨论视觉编码器，FALCON 讨论动作头，R3DP 讨论异步频率；这些问题叠在一起，才构成一个真正可部署的 geometry-conditioned VLA。换句话说，未来最强的方法很可能不是“某一层加 3D 最优”，而是一个多层接口协同方案：前端少量几何对齐，中段上下文控制融合，末端动作头强化，再加异步慢系统定期校正。这个推断是我基于这几篇的共同趋势做的归纳。

如果再往“你自己的项目还能怎么做”这一步走，我会说，最值得继续推的已经不是“再找一个新的 3D feature source”，而是研究 几何先验的调度机制。比如说，哪些时刻该更多信 3D，哪些时刻该更多信语义；几何 token 是否应该随 task phase 改变注入深度；慢系统的几何校正是否可以只在 uncertainty 高的时候触发；scene flow、point flow、depth token、camera-aware feature 这些不同表征，是否应该在不同动作阶段扮演不同角色。前面这些论文已经把“3D 有用”证明得差不多了，下一步真正有原创空间的，是3D 如何被按需调用、按层调用、按阶段调用。这一点我觉得就是你能把项目做大的地方。

如果你愿意，我下一条可以继续把这六篇压成一张更实用的“研究地图”：我直接帮你总结成 三个可做的原创题目方向，每个方向都写清楚适合借鉴哪几篇、最合适的 baseline、核心创新点该落在哪。

详细展开昨天说的第二点 ​

3D 运动先验放在了哪里、怎么进策略、为什么比单纯几何融合更进一步。

motion-prior VLA 更进一步：机器人做动作时，模型内部有没有一个显式的世界接下来会怎么动的中间变量。LaMP 和 SG-VLA 都在回答：不要把动作完全建立在 2D 语义 token 上，而要强迫模型先形成一个和运动、接触、位姿、抓取状态有关的 bottleneck，再去出动作。 只是 LaMP 选的是 未来运动先验 ，SG-VLA 选的是 当前空间状态先验。现有很多 VLA 直接从 2D 语义视觉特征回归动作，这等于逼模型把复杂的 3D 物理交互——谁会动、往哪动、深度怎么变、什么时候发生接触——都隐式塞进 action supervision 里去学；这种学法在训练分布内可以凑合，但一旦遇到新视角、布局变化、精细深度控制，性能就会脆。SG-VLA 站在另一个场景下也得出类似结论：移动操作里的动作空间更高维，既要底盘导航又要手臂操作，单靠“视觉+语言→动作”的直接 imitation，监督太稀薄，模型很难学出真正 manipulation-aware 的表示。两篇工作的共同起点，其实都是在否定“end-to-end 直接映射就够了”这个默认前提。

LaMP ​

把 3D scene flow 当成 latent motion prior。 LaMP 的思想：动作之前，先让模型内部产生一个“未来场景怎么动”的稠密 3D 运动隐变量，然后再让这个隐变量去条件化 action generation。 它不是把 3D scene flow 当成最终输出，也不是做一个独立的 world model 再外接 policy，而是把 scene flow 作为 VLA 内部的 latent motion prior。作者显式把 joint distribution 分解成“先建模 motion，再在 perception + motion 条件下建模 action”，也就是把动作预测写成一个带中间变量的两步过程。理论直觉是：action 不应直接从语义图像特征里“跳出来”，中间最好经过一个与未来动态相关的结构化变量。

LaMP 选的中间变量不是 2D optical flow、 sparse trace，而是 dense 3D scene flow。它在当前图像平面上采样规则网格点，跟踪这些点在未来若干步里的位移增量；这个位移不是纯 2D，而是“图像平面位移 + 深度变化”，因此保留了显式 3D 几何。这种表示既比 2D flow 更有物理意义，又比稀疏轨迹更有 scene-level consistency；同时它还把轨迹统一到参考相机坐标系里，减少相机运动带来的扰动，并保持和 2D 轨迹格式的兼容性，方便跨数据源训练。

在架构上，LaMP 是一个 dual-expert 设计。第一个 expert 是 Motion Expert，本质上是一个在 motion-token 空间里运行的 3D transformer，风格上接近 CogVideoX 类的生成模型；它用 conditional flow matching 来学从噪声到真实 3D scene flow 的速度场，并且用最后一层 VLM hidden states 作为条件输入。第二个 expert 是 Action Expert，它不再只看原始 VLM 特征，而是看“被 motion prior 引导过”的特征，再用 flow matching 生成连续动作。换句话说，LaMP 不是“用 VLM 预测动作，再额外预测一下 flow”，而是让 motion generation 先走一步，action generation 后走一步。

它怎么把 scene flow 引进 action policy 比较巧妙。如果先把完整的 3D scene flow 全部 denoise 干净，再拿去喂 policy，代价会很高，也会把动作链路拖慢。所以 LaMP 没这么做。它只从纯噪声出发，跑 一步 partial denoising，拿到一个“还没完全干净，但已经含有任务相关动态信息”的中间 motion hidden states。作者明确说，这样既保留了动态线索，又绕开了 full reconstruction 的计算开销和偏差。这里的思想其实和一些视频条件 policy 的 partial denoising conditioning 很接近：不求未来完全生成出来，只求生成一个足够有用的中间动态隐状态。

然后，这个 motion hidden state 不是暴力加进所有层，而是通过一个 gated cross-attention 在最后一层 VLM 特征上做单层注入。这个 gate 从接近 0 开始学，也就是一开始几乎不让 motion prior 影响原有 VLM 表示，等训练发现 motion guidance 真有用时再逐渐放大。作者明确说他们发现 naive fusion 会导致 representational collapse，所以必须用这种“弱注入起步、按需增强”的策略。这个点很重要，因为它说明 LaMP 的真正创新不是“future dynamics + policy”这么简单，而是它找到了一个不破坏预训练 VLM 语义能力的注入接口。从研究 sense 看，这比 scene flow 本身还重要。

训练上，LaMP 采用两阶段。第一阶段只训练 Motion Expert，用 TraceForge 生成的 dense 3D scene-flow supervision 来做 motion-prior pretraining；论文说他们构建了 160 万条 observation-language-motion triplets，来源包括 LIBERO、BridgeV2、DROID 和 InternData-A1。第二阶段冻结 Motion Expert 和 VLM backbone，只训练 Motion Guidance 模块和 Action Expert；训练和推理时都使用同样的一步 partial denoising 过程来抽 motion guidance feature。这个训练设计很干净：motion prior 先学会，再作为稳定的动态教师去引导动作。

实验上，LaMP 的结果也非常符合它的设计目标。它在 LIBERO、LIBERO-Plus 和 SimplerEnv-WidowX 上都优于评测到的 VLA 基线；在 LIBERO-Plus 的 zero-shot OOD 上，平均成功率达到 79.3%，比文中最强先前基线 OpenVLA-OFT 的 69.6% 高 9.7 个百分点。在 SimplerEnv-WidowX 上，它平均成功率 79.2%，比文中最强基线 FlowVLA 的 74.0% 高 5.2 个百分点；作者还特别指出，在更难的 Stack Block 任务上优势更明显。更关键的是消融：去掉 Motion Expert 后，SimplerEnv-WidowX 平均从 79.2% 掉到 56.3%，说明这个 motion prior 不是装饰，而是核心贡献。论文还展示了真实机器人实验。

所以如果把 LaMP 一句话说透，它真正做的不是“给 VLA 加一个 3D 模态”，而是：把“未来场景如何运动”的显式猜想，变成动作生成前的内生中间层。 这就是为什么我前面说它属于 motion-prior VLA，而不只是 geometry-aware VLA。geometry-aware 解决的是“物体在哪”；LaMP 更进一步，开始解决“物体接下来会怎么动、机器人应该顺着什么动态去动作”。这个层级已经更接近小型 world action model 了。这个结论是我基于论文方法作的归纳。

SG-VLA ​

不显式预测未来流，而是把“空间状态”拆成可监督的中间变量。 SG-VLA 没有去建 future scene flow，也没有显式 world model。它的基本判断是：移动操作里 VLA 学不好，不一定是因为缺未来预测，也可能是因为当前状态表征本身就没学扎实。于是它用两种方式去“压出”一个 manipulation-aware 的空间中间层：一是给模型更丰富的空间输入，二是让模型必须从共享 latent 里重建一组和操作密切相关的中间量。

第一部分是输入增强。SG-VLA 明确采用 multi-view RGB、depth cues 和短 temporal history。在具体实现上，它是一个 1.3B 参数的轻量模型，主体是预训练的 Prismatic VLM 骨干，视觉侧采用 DINOv2 + SigLIP 双编码器，语言侧用 Qwen2.5-0.5B，外加 projector 做视觉—语言对接；还可以选择接一个 100M 的 flow-matching action expert 生成连续动作。你可以看到，这个框架本身并不激进，真正激进的是它对“中间监督”的使用。

第二部分是 SG-VLA 的核心，也就是一组 auxiliary decoders。它让共享的视觉—语言 latent 不只负责动作，还同时去预测当前时刻的一系列可解释中间信号：包括 机器人全局位置、是否已抓住物体、目标物相对位姿、关节配置 qpos、目标物分割 mask 等等。为了适配不同目标，它用了不同类型的 decoder：低维向量用 MLP，高维且有关节依赖结构的 qpos 用 transformer，mask 用 CNN。然后用总损失把 action loss 和这些 auxiliary losses 一起优化。这个设计背后的想法很明确：如果一个 latent 真正掌握了 manipulation 所需的空间状态，它就应该能把这些中间变量解码出来；如果解不出来，说明它的“理解”还不够结构化。

这里你可以看到它和 LaMP 的根本差异。LaMP 是在建模“未来会怎么动”，SG-VLA 是在强迫模型把“当前世界处于什么空间状态”说清楚。一个是 future dynamics bottleneck，一个是 current-state bottleneck。但两者精神完全一致：不要让 action head 自己从混杂的 2D 语义特征里瞎悟 manipulation，本体论上先给它一个更结构化的中间表示。

SG-VLA 还有一个很值得注意的点，是它非常重视训练稳定性。如果把随机初始化的 auxiliary decoders 和 VLM backbone 从一开始就一起训，会出现明显退化，因为这些 decoder 会向预训练 backbone 打入很吵的梯度。于是他们采用 progressive multi-stage training。先是 decoder adaptation 阶段：阻断 auxiliary decoder 到 backbone 的梯度，只让 decoders 先学会“读懂”固定的 VLM latent；然后再开全梯度 joint refinement，让这些辅助任务真正反向塑造 backbone 表示；最后如果要用 flow-matching action head，再把 backbone 冻住，单独训练 action head。这个训练 scheme 很像是把“多任务共训练”做成了一个 curriculum，而不是一次性硬训。

实验上。论文在 ManiSkill-HAB 的 household mobile manipulation 任务上报告，SG-VLA 的平均成功率达到 73%，而同架构上的直接 imitation learning 基线是 60%。输入模态消融里，多视角已经有明显收益，在此基础上加入 depth 又从 0.52 提到 0.60；反而加入短时历史后降到 0.49，说明“历史”不是加了就有用，模型未必能有效整合。更关键的是辅助任务消融：不做 progressive training 时，把全部 auxiliary 任务一起上反而从 0.60 掉到 0.51；做了 progressive training 后，再把 grasp、qpos、global position 全部加上，可以升到 0.73。而在跨任务的 pick/place 场景中，再加入 segmentation 和 object position reconstruction，平均还能从 0.27 提到 0.47。这些结果说明 SG-VLA 的增益不是来自某个单一 fancy module，而是来自“把 manipulation state 拆成多个可解释 supervision channel”。

论文里还有一个很有意思的结论：加入 flow-matching action head 不一定总是更好。总平均从 0.73 反而降到 0.69，虽然在 pick 上有明显提升。这意味着 SG-VLA 真正有效的部分，并不一定是更复杂的动作生成器，而是前面的 state grounding。这点和 LaMP 很不同：LaMP 的强项就是把动态先验显式地送进 Action Expert；而 SG-VLA 的结果更像在说，只要状态表征足够扎实，未必要靠更强 generative action head 才能提升整体性能。 这个现象很值得你记住。

主线是什么 ​

如果把 LaMP 和 SG-VLA 放在一起，你会发现它们刚好构成了 motion-prior VLA 里两种非常典型的路线。

LaMP 的路线是：先学一个 future dynamics prior，再让它条件化动作。 它更接近“轻量 world model inside policy”。所以它特别擅长处理 OOD 扰动、跨视角、跨 embodiment、长时序、多阶段 manipulation 这类问题，因为这些问题本质上都要求模型心里先有一个“未来会怎么展开”的猜想。

SG-VLA 的路线是：先把 current state 拆成多个 manipulation-relevant 中间变量，再让动作建立在这个空间化 latent 上。 它更接近“structured state supervision inside VLA”。所以它特别适合移动操作、复杂 household environment、导航和手臂联合控制，因为这些任务的困难点往往是状态因素太多、监督太稀，不一定非要显式 rollout future 才能受益。

换句话说，LaMP 和 SG-VLA 的差别，不是“一个用了 3D flow，一个没用”；真正的差别是：LaMP 给动作头一个未来运动瓶颈，SG-VLA 给骨干表征一个当前状态瓶颈。 前者偏 dynamics，后者偏 state estimation；前者偏“想象世界如何变化”，后者偏“明确世界现在是什么样”。这两者其实不是对立的，反而很可能是可以叠的。这个组合判断是我的推断，但和两篇论文的方法结构是对齐的。

真正的创新点 ​

LaMP 真正的创新点，我认为有四层。第一层是它把 dense 3D scene flow 从“单独 world model 的输出”变成了 VLA 内部 latent prior，这在论文中被明确当作主要贡献。第二层是它没有走 full scene-flow rollout，而是用 one-step partial denoising 提供足够有用但代价更低的动态中间态。第三层是它发现了 naive fusion 的问题，用 zero-init gate + cross-attention 解决 motion prior 注入时对 VLM 的破坏。第四层是它把 motion prior pretraining 和 policy learning 明确分阶段，构成一个相对可扩展的 pipeline。

SG-VLA 真正的创新点也不只是“多视角+depth+auxiliary”。第一，它的贡献在于把 mobile manipulation 的空间需求 正式放进 VLA 训练目标里，而不是延续桌面抓取那套简单范式。第二，它提出的不是随便几个 auxiliary loss，而是一组和机器人实际操作 strongly aligned 的中间信号：qpos、grasp、global pose、object pose、segmentation，这些变量分别约束了机器人本体状态、目标物状态和场景局部几何。第三，它非常认真地处理了 co-training stability 问题，用 progressive training 证明辅助监督不是越多越好，而是必须以正确方式注入。第四，它揭示了一个很有研究价值的现象：对某些任务，状态表征的改善比更复杂的 action head 更重要。

比单纯 geometry fusion 更有意思 ​

因为 geometry fusion 解决的往往只是“看懂 3D”，而这两篇开始碰的是 manipulation intelligence 的核心变量。真正难的 manipulation，不是知道杯子在桌上，而是知道你手一推它会不会滑、抓上去会不会掉、开抽屉时本体该如何配合、挡住视线时接下来哪个局部会发生接触。LaMP 直接把“世界接下来如何动”显式化了；SG-VLA 则把“做动作时必须知道的状态因素”显式化了。它们都在把“动作产生之前模型内部该拥有什么结构”这个问题说清楚。这个问题，比“3D token 要加在第几层”要更本质。

sense ​

我觉得最值得你抓住的是，不要把 future motion prior 和 current-state grounding 看成两条完全不同的线。更可能的长期方向是：先用 SG-VLA 风格的辅助任务，把当前空间状态学扎实；再用 LaMP 风格的 motion prior，在这个更干净的 latent 上建短时未来动态。 也就是说，一个负责回答“现在是什么”，另一个负责回答“接下来会怎么变”。如果两者能在同一个 backbone 上协同，那个中间层就不只是 geometry-aware，也不只是 motion-aware，而是一个真正可用于 manipulation 的 state-dynamics bottleneck。这部分是我的推断，不是论文原文结论，但我觉得它正是你后面可以做成项目主线的地方。

另外，LaMP 自己也承认当前 Motion Expert 仍然受固定空间分辨率和固定时间 horizon 限制；SG-VLA 的结果则显示，历史信息和 flow-matching action head 并不是天然带来收益，说明“时间建模”本身还没有被它完全解决。也就是说，两篇都很强，但各自缺一半：LaMP 强在动态，SG-VLA 强在状态。正因为如此，这个方向还有很大空间。

详细展开昨天说的第三点 ​

世界状态怎么表示、动作怎么表示、未来怎么预测、怎么接控制。这样你能直接看出它们各自押注的路线差异。

把 ParticleFormer、PointWorld、DreamZero 放在一起看，这个判断会特别清楚，因为它们分别代表了三种很典型的答案：ParticleFormer 是“用显式 3D 点云动力学做世界模型，再接 MPC”；PointWorld 是“把 state 和 action 都统一到 3D point flow 里，直接做可扩展的大模型式 3D world model”；DreamZero 则是“虽然不显式建 3D 点云，但把 video 作为世界演化的密集载体，联合预测未来视频和动作，让 video world model 直接变成 policy”。三篇风格不同，但都在把“动作生成”往后推，把“世界演化建模”往前提。

先把这条线和普通 VLA 区分清楚。普通 VLA 的默认范式是：给当前观察和语言指令，直接回归动作。它当然也会在内部隐式学一点物理，但物理只是为动作服务的隐变量。显式 world model / WAM 的逻辑正好相反：先学“如果机器人这样动，世界会怎么变”，再利用这个预测去规划或选动作。ParticleFormer 明确把 world model 定义成 learning-based 3D dynamics model；PointWorld 直接把任务写成“给 RGB-D 和动作，预测全场景 3D point flow”；DreamZero 则说得更彻底，它认为 video prediction 本身就是 implicit visual planner，动作应该和未来视频一起生成。也就是说，这条线里 action 不再是唯一主角，真正的主角是 action-conditioned future world state。

ParticleFormer ​

把 3D world model 从“单一材料 + 图网络”推进到“多材料 + Transformer 点云动力学”。 ParticleFormer 的起点非常明确：以前很多 3D dynamics model，尤其是 particle-based 那一支，常常依赖 graph-based neural dynamics。这类方法需要先定义粒子邻接关系，对最大邻接距离、邻居数这些超参数很敏感；一旦场景里同时有 rigid、deformable、flexible 多种材料，接触关系异质而复杂，这种固定拓扑就会变得脆弱。更麻烦的是，很多方法为了训练 3D 粒子动力学，还要先做耗时的动态 3D 重建，比如 dynamic Gaussian Splatting，才能拿到 3D particle tracks。ParticleFormer 就是冲着这两个瓶颈来的：它要把 3D world model 从“图结构 + 重建依赖”解耦出来。

它的方法可以概括成两个核心设计。第一个是 Transformer-based neural dynamics backbone。论文在方法部分把整个模型拆成 observation embedding、dynamics transition、motion prediction 三个模块，用 Transformer 的多头自注意力直接学习粒子级交互，而不是事先写死图连接关系。这样一来，模型自己决定哪些点之间要发生强耦合，不再过度依赖手工设定的局部拓扑。这个改变看起来像是“把 GNN 换成 Transformer”，但本质上是在换掉世界模型的归纳偏置：从“先验规定谁和谁相互作用”，变成“让模型从数据里自己发现 interaction law”。对 multi-object、multi-material 交互，这一步尤其重要。

第二个核心设计是它的 hybrid point cloud loss。ParticleFormer 不再要求点与点的严格 correspondence，也不依赖昂贵的显式 3D scene reconstruction，而是直接在 point cloud sensor stream 上做监督。具体上，它把 Chamfer Distance 和一个可微的 Hausdorff Distance 近似结合起来：Chamfer 负责局部平均对齐，Hausdorff 负责约束最坏情形下的大偏差。作者专门解释了为什么单用 Chamfer 不够——在多材料场景里，Chamfer 容易把训练重心放到那些运动幅度大的区域，忽略被间接作用到、但语义上同样重要的“被动点”；Hausdorff 正好能把这种最坏偏差抓回来。换句话说，这个 loss 不是普通的“几何重建损失”，而是在显式逼模型同时学 局部精细运动 和 全局结构一致性。

这篇论文的方法论价值在于，它把 3D world model 的研究重点，从“怎样把粒子系统拟合得像物理仿真器”转向了“怎样从真实感知数据里直接学可用的 3D interaction model”。所以你会看到它强调自己直接从真实机器人感知数据训练，避免 elaborate reconstruction；同时它又能把预测结果直接接进 MPC 做下游控制。也就是说，ParticleFormer 不是只想证明“我能 forecast 点云”，而是要证明“我的点云世界模型足够好，已经能支撑 model-based control”。这也是它和很多纯预测论文的本质区别。

如果再往深一点看，ParticleFormer 实际上在回答一个很关键的问题：为什么显式 3D 仍然有必要？ 它给出的答案不是“因为 3D 更直观”，而是“因为 2D latent world model 把整个场景压成一个向量后，容易丢掉 compositional scene 和 fine-grained dynamics 所必需的结构”。这点你如果做 manipulation，会很有感触：绳子带动盒子、布料拖拽、颗粒流动、柔体和刚体接触，这些现象不是一个小 latent vector 很容易讲清楚的。ParticleFormer 的贡献，就是把这类 interaction dynamics 放回点云空间里，用更少手工假设的方式学出来。

但它也有明显边界。虽然它已经比老的图动力学模型灵活很多，但它仍然属于比较“经典”的显式 3D world model：输入是点云，输出是点云未来；控制靠 MPC；重点是 dynamics accuracy 和 rollout error。它还不是大规模预训练意义上的 foundation-style world model，也没有像 PointWorld 那样系统谈 scaling laws、跨 embodiment 统一动作空间，或者像 DreamZero 那样直接把 generative model 做成 policy。你可以把 ParticleFormer 看成：这条线从“能不能建显式 3D world model”走向“能不能把它做好到足以支撑复杂多材料操控”的关键一步。

PointWorld ​

把 state 和 action 统一到同一个 3D 空间里。 如果说 ParticleFormer 还是“经典 3D dynamics model 的强化版”，那 PointWorld 就明显更像这条路线进入 foundation model 阶段的标志。它最核心的 idea 其实一句话就能概括：state 和 action 不应该分属两个空间，而应该统一表示成同一种 3D 几何对象——point flows。论文里说得非常直接：state 用 RGB-D 构建的 full-scene point cloud 表示，action 则不再是 joint positions 或 end-effector command，而是由机器人自身几何体在时间上的 3D point trajectories / point flows 表示。于是 world modeling 就变成了：给定当前场景点云和机器人点流，预测场景里每个点如何位移。

这个设计特别强，因为它同时解决了两个老问题。第一个问题是 动作空间的 embodiment specificity。过去很多 manipulation 模型，Franka 的关节命令和 humanoid 的动作表示根本不是一个空间，很难一起学。PointWorld 通过把机器人动作转换成“机器人表面点在未来几步里怎么移动”，把动作放回了几何空间。论文明确说，这样做可以在单臂、双臂、whole-body、mobile manipulation 等 heterogeneous embodiments 之间实现更自然的共享学习。第二个问题是 接触建模。如果动作只是一个低维向量，模型必须自己再去猜“这个向量意味着哪个手指、哪个表面、在哪发生接触”；而机器人点流把 interaction geometry 直接暴露了出来。

PointWorld 的输入构造很有意思。它从 calibrated RGB-D 得到 scene points，再用机器人 URDF、forward kinematics 和 proprioception，把关节空间动作转换成 robot flows；然后把 scene points 和 time-stacked robot points 拼成一个大的 point cloud。场景点用冻结的 DINOv3 特征做几何语义增强，机器人点则带 temporal embedding，再交给 point cloud backbone 去预测未来若干步的 full-scene point flow。这里它没有自己发明一套奇怪的新网络，而是刻意站在现成强点云骨干上，比如 PointTransformerV3，去研究“什么 recipe 能让 3D world model 真正 scale up”。这点非常像 foundation model 研究的做法。

这篇论文真正厉害的地方，不止是表示统一，而是它非常认真地做了 scaling study。论文反复强调自己不是只提一个模型，而是在大规模数据上系统研究了 backbone、action representation、objective、partial observability、data mixtures、domain transfer 和 scaling laws。它训练数据来自真实和模拟机器人 manipulation，覆盖单臂 Franka 和双臂 humanoid，总量大约 200 万条轨迹、500 小时。作者甚至明确写到，随着模型规模从 50M 到 1B、数据量从 5% 到 100% 增加，预测误差在 log space 里呈现近似线性下降。这意味着 PointWorld 不只是“提出了一个好点子”，而是在证明：显式 3D world model 也可以像语言模型或视觉模型那样，进入可预测的 scaling regime。

从 world model 角度看，PointWorld 的另一个关键创新是它把监督做成了 dense pixel/point-level 3D flow supervision。真实数据里，这套监督依赖深度估计、相机位姿估计、点跟踪，把 2D tracks lift 成 3D scene flows；机器人自己的 flow 则由 URDF 和运动学直接得到。对 real-world data，论文还专门做了 visibility-aware supervision，只在可见、深度有效的点上计算损失，避免被遮挡和坏深度污染。这说明它很清楚大规模真实 3D 标注的噪声问题，不是“有了大数据就自动有效”，而是必须先把 annotation pipeline 做扎实。

再说控制接口。PointWorld 不是学完预测就停，而是明确把自己嵌进 sampling-based MPC / MPPI。因为它一次前向就能预测一个短 horizon chunk 的 scene point flows，推理延迟大概是 0.1 秒，所以可以高效评估大量候选动作序列。论文在真实 Franka 上展示了 pushing、deformable manipulation、articulated object manipulation 和 tool use，而且强调是单个预训练 checkpoint、单张 in-the-wild RGB-D 图像、无 demonstrations、无 post-training。这一点的意义很大：它说明显式 3D world model 不必只是“分析工具”，它已经可以在 zero-shot 或 near-zero-shot 条件下成为动作生成引擎。

但 PointWorld 的边界也要看清。论文自己承认，它当前假设 初始时刻世界是静态的，因为输入里没有 prior trajectories 或 recurrent state；支持 fully dynamic initial conditions 需要额外的历史跟踪或循环状态，这是未来工作。换句话说，PointWorld 已经很强，但它本质上还是一个 short-horizon, action-conditioned, static-initial-state 3D dynamics model。它比 ParticleFormer 更大、更统一、更可扩展，但仍然不是一个长期记忆型的开放世界 3D simulator。这个限制对你后面想项目时很关键，因为它提示了下一步最自然的研究方向：如何把显式 3D world model 从“静态瞬时观测 + 短期 rollout”推向“带 persistent state 的持续世界建模”。

DreamZero ​

不做显式 3D 点云，但把 video world model 直接变成 zero-shot policy。 DreamZero 和前两篇最大的不同，是它不走显式 3D 点云世界模型，而是把 video diffusion world model 直接变成 action policy。论文自己把它叫 World Action Model (WAM)，核心主张是：相较于 VLA 直接学 observation-to-action mapping，WAM 应该联合建模 future video + future action，让视频成为“世界如何演化”的密集表示。DreamZero 的作者甚至明确写道，video prediction 在这里充当的是 implicit visual planner，动作则由这个未来视觉计划来引导。也就是说，在 DreamZero 里，policy 已经不是单独的一条头，而是 world model 的一个伴生产物。

方法上，DreamZero 建在一个很大的预训练 video diffusion backbone 上。论文写的是 14B autoregressive diffusion transformer，训练目标是 teacher-forcing 的 chunk-wise video denoising objective。它把输入拆成三部分：视觉上下文通过 VAE 编码，语言指令通过 text encoder 编码，proprioceptive state 通过 state encoder 编码；这些信号一起送进 autoregressive DiT backbone，用 flow matching 同时去预测未来视频帧和未来动作。这里最关键的，不是“大模型”三个字，而是它坚持了 joint denoising / joint modeling：视频和动作不是两个松散的头，而是在同一个生成过程中一起被去噪、一起被预测。

为什么 DreamZero 强调 autoregressive，而不是 bidirectional 或固定长度序列？论文讲得很清楚。作者认为 autoregressive generation 有三个好处：一是可以利用 KV cache 提高推理速度；二是能自然利用更长的历史上下文；三是更容易保持视频帧率和动作时序的严格对齐，而不用像 fixed-length bidirectional diffusion 那样为适配序列长度去做 subsampling，破坏原生 FPS。换句话说，DreamZero 并不是简单把 video diffusion 搬到机器人上，而是在认真解决“生成式世界模型怎么和控制频率对上”的系统问题。

DreamZero 还有一个很聪明的闭环设计：每执行完一个 action chunk，就把真实观测重新注入 KV cache，替换掉预测视频帧。论文明确说，这样能在 closed-loop setting 下阻止误差累积。这个想法很重要，因为它体现了 video WAM 和一般 world model rollout 的一个核心区别：DreamZero 不打算让自己在 open-loop 下无限长地“做梦”，它利用真实机器人每一步都能给到新观测的事实，选择在每个 chunk 之后用 ground-truth observation 把 imagined future 拉回现实。这其实是一种很实用的“模型想象 + 真实校正”折中。

从结果上看，DreamZero 最突出的地方不是单个任务指标，而是它在 泛化和迁移 上展示出的特性。论文在 AgiBot G1 上报告，DreamZero 在 seen-task zero-shot environment generalization 上平均 task progress 达到 62.2%，相对文中最强 pretrained VLA baseline 的 27.4% 超过 2 倍；在完全未见任务上，DreamZero 平均 39.5%，对比 pretrained VLA baseline 的 16.3% 也有很明显优势。它还展示了 cross-embodiment transfer：只用 10–20 分钟 的其他机器人或人类 video-only demonstration，unseen task performance 相对提高超过 42%；对全新 embodiment 的 few-shot adaptation 只用了 30 分钟 play data。

更有意思的是 DreamZero 对“为什么有效”的解释。论文明确说，大模型 video generation 质量和 downstream action execution 质量高度相关；甚至很多失败并不是 action decoder 自己乱来，而是 video prediction 错了，policy 忠实地执行了错误的未来轨迹。这句话其实非常有分量，因为它说明在 DreamZero 这个范式里，policy quality 被上移成了 world-model quality 的函数。换句话说，提升机器人能力，某种意义上就等价于提升视频世界模型的预测质量。这已经不是“world model 帮 policy”了，而是“policy 被 world model 吞进去”。

当然，DreamZero 和 ParticleFormer、PointWorld 的哲学差得很大。前两者押注显式 3D 世界状态，DreamZero 押注视频作为世界演化的密集表征。DreamZero 的好处是天然能吃到 web-scale video diffusion prior，也更方便接 heterogeneous robot data 和 video-only data；但它的物理状态不是显式 3D 点集，所以在需要精确 geometry、contact attribution、cost-in-state-space planning 的任务里，不像 PointWorld 那样“状态是什么就摆在那儿”。论文自己也承认，附录里专门讨论了 WAM 和 3D point cloud world model 的区别。你可以把 DreamZero 看成：它不是显式 3D 世界模型，但它代表了另一种同样很强的世界动力学中心主义——让视频世界模型直接成为 policy。

分叉 ​

如果把三篇压缩成一个统一视角，最本质的区别其实只有两个维度。第一个维度是：世界状态显不显式。ParticleFormer 和 PointWorld 都是显式 3D；ParticleFormer 更偏经典 object/particle-centric 3D dynamics，PointWorld 更偏 scalable, foundation-style 3D point-flow modeling。DreamZero 则是隐式世界状态——它让 future video 代替显式 3D state，动作跟视频一起生成。第二个维度是：动作是附属条件，还是和状态统一建模。ParticleFormer 里动作主要还是给 dynamics model 当条件；PointWorld 把动作本身变成和 state 同空间的 3D point flow；DreamZero 则把动作和视频一起作为 joint generative target。

因此它们三个其实对应三种越来越激进的研究姿态。ParticleFormer 说的是：先把世界动力学学准，policy 可以后接。PointWorld 说的是：state 和 action 要在表示层统一，世界模型才能真正跨 embodiment、跨任务 scale。DreamZero 说的是：既然未来视频已经能承载规划信息，那 policy 干脆就从 world model 里长出来。 你会发现，这正对应了我前面那句话的三个强度版本：policy 先是“退到 world model 后面”，然后“和 world model 的表示统一”，最后“被 world model 直接吸收”。

因为一旦你认真看这三篇，就会发现它们在训练目标上已经和普通 VLA 根本不是一个问题了。普通 VLA 的损失函数主轴始终是 action prediction；就算加一些 auxiliary losses，最终也大多是为了让 action head 更强。而在这三篇里，真正被当成一等公民优化的是 future world evolution。ParticleFormer 优化的是 point-cloud dynamics 及其 rollout 误差；PointWorld 优化的是 action-conditioned full-scene 3D point flow；DreamZero 优化的是 joint video-action denoising / flow matching。它们当然都要出动作，但动作不再是唯一监督对象，而是和“世界怎么变”并列，甚至从属于后者。

这件事为什么重要？因为 manipulation 真正难的地方，从来都不是“识别杯子”这种语义问题，而是 接触、遮挡、形变、铰接、工具、力的传递、局部不可见处会发生什么。这些东西如果不通过世界动力学来建模，policy 最多只能在数据分布里学一些统计共现。一旦换任务、换视角、换物体、换 embodiment，就容易崩。DreamZero 报告的强泛化、本质上来自世界演化建模；PointWorld 在真实机器人上从单张图做 zero-shot MPC，本质上也依赖于它学到了 transferable interaction dynamics；ParticleFormer 能同时处理绳子、布、颗粒、盒子混合场景，也是因为它把世界建模问题正面提了出来。

sense 看 ​

ParticleFormer 最值得学的，不只是“Transformer 比 GNN 强”，而是它告诉你：多材料、多物体的显式 3D world model，关键不在对象识别，而在 interaction representation 和 supervision design。 它用 Transformer 摆脱手工图拓扑，用 Chamfer + Hausdorff 摆脱 correspondence / 重建依赖，这两步都直指“如何从真实点云里稳定学到可控的动力学”。

PointWorld 最值得学的，是它的 unification for scaling。也就是它不把 state、action、embodiment 分开想，而是问：有没有一个共同空间，能把机器人身体和外部世界都放进去？它的答案是 3D point flow。这个统一视角很可能比某个具体 backbone 更重要，因为它直接决定了模型能不能跨 embodiment、跨数据源一起学。再加上它认真研究了 scaling、partial observability、data mixture，这使它很像“3D world model 版的大模型 recipe paper”。

DreamZero 最值得学的，是它把 policy = world model 的一个使用方式 这件事说透了。它的核心创新不是单纯“大视频模型控制机器人”，而是 joint video-action modeling、autoregressive chunked inference、真实观测回灌 cache、防止误差累积，以及一整套把 14B video diffusion 压到 7Hz 的系统优化。换句话说，DreamZero 最大的创新点不是某个 loss，而是它把“世界模型中心主义”做成了一个真正能闭环运行的系统。

追问 ​

我觉得这三篇放在一起，给出的最自然研究问题不是“谁最好”，而是三条更具体的主线。第一条是 显式 3D state 和 video latent 到底该怎么结合。ParticleFormer / PointWorld 强在可解释 geometry，DreamZero 强在大规模 generative prior 和跨 embodiment transfer，这两边迟早会融合。第二条是 短时 3D dynamics 和长时 persistent state 怎么打通。PointWorld 现在还是 static initial state + short horizon，ParticleFormer 更偏局部 rollouts；如果你想做更大的系统，这里就是突破口。第三条是 world model 和 policy 的边界还要不要保留。PointWorld 还保留了 MPC 这一层，DreamZero 已经在弱化它；未来很可能出现一种折中：前端是显式 3D world model，后端是 learned planner / action decoder，而不是纯手工 MPC，也不是纯 observation-to-action policy。这个判断是我基于三篇方法结构做的推断。

所以，回到你最初那句话，我会把它再压成一句更学术一点的话：这条线的真正转向，是把机器人学习的中心目标从“模仿动作分布”转成“学习动作条件下的世界演化规律”，而 policy 只是对这些规律的一种读出方式。 ParticleFormer 是把这件事做成显式 3D 点云动力学，PointWorld 是把它做成可扩展的统一 3D point-flow foundation model，DreamZero 是把它做成 joint video-action generative policy。三篇合起来，基本就是 2025–2026 这条路线最核心的三种答案。

思考 ​

早在 pointvla 中就提到过 3d 数据规模远小于 2d 机器人数据，硬改肯定会破坏学会的2d表征，pointvla 做了一个 skip-block analysis：在 DexVLA 的 32 个 diffusion transformer blocks 里，逐个或连续跳过某些 block，看性能掉不掉。发现前 11 个 block 很关键，跳了会明显坏掉；但从第 11 个以后，部分 block 在推理阶段被跳过，性能影响没那么大，甚至最多可以连续跳过 5 个 block 才彻底失败。于是他们就把 3D 特征选择性地注入到这些“相对没那么关键”的 block 里。这个设计把点云当作对后段动作生成过程的几何修正。
